# SECURITY

Smart Barber Shops uses a practical cloud and application security baseline aligned to ISO/IEC 27001-style governance intent and informed by ISO/IEC 27002-style control guidance. It is not formally certified.

## Security Baseline

- Least-privilege managed identities for runtime, GitHub infrastructure automation, and GitHub deployment automation
- Azure Key Vault for secret storage and runtime secret resolution
- GitHub Actions authentication to Azure through OIDC workload identity federation instead of long-lived client secrets
- Separate `dev` and `prod` environments with distinct naming, scaling, GitHub environments, and managed identities
- Infrastructure defined in Terraform and reviewed through workflow-based plan/apply flows
- Azure Container Apps for the web runtime and a separate migration job to avoid implicit schema changes on every scale event
- Auditability through GitHub Actions logs, Azure Activity Logs, Key Vault access, and Log Analytics application/runtime logs
- Django runtime protections already implemented in the application layer, including CSRF, security headers, session controls, and audit logging

## Secrets Handling

- `DJANGO_SECRET_KEY` and the PostgreSQL admin password are generated by Terraform and written into Key Vault.
- Runtime secret resolution is handled through Key Vault secret references in Container Apps using the runtime managed identity.
- Secrets are not committed to source control and are not stored in GitHub repository secrets for Azure login.
- Tradeoff: Terraform state contains generated secret values. Remote state must therefore be encrypted, access-controlled, and backed by a hardened Azure Storage account.

## Identity Model

- Runtime identity:
  - `AcrPull` on the ACR
  - `Key Vault Secrets User` on the Key Vault
- GitHub infra identity:
  - `Contributor` on the environment resource group
  - `User Access Administrator` on the environment resource group
  - `Key Vault Secrets Officer` on the Key Vault
- GitHub deploy identity:
  - `Container Apps Contributor` on the environment resource group
  - `AcrPush` on the ACR

Detailed OIDC configuration is documented in [docs/github-oidc-setup.md](/home/khido/projects/barbershop/docs/github-oidc-setup.md).

## CI/CD Trust Model

- GitHub Actions jobs request short-lived OIDC tokens.
- Azure trusts only the configured GitHub repository and environment subjects.
- `prod` deployments are intended to be gated by GitHub environment approvals.
- No publish profiles or long-lived Azure passwords are required for the workflows in this baseline.

## Infrastructure and Network Posture

- Public ingress is enabled for the web application through Azure Container Apps.
- PostgreSQL Flexible Server uses public networking with the Azure-services firewall allowance enabled for the MVP.
- TLS is expected end-to-end for the public application URL.
- Key Vault uses RBAC authorization and purge protection by default.
- Log Analytics provides centralized platform/application diagnostics.

## Residual Risks

- Terraform state contains generated secrets and therefore becomes a sensitive asset.
- PostgreSQL public network access with the Azure-services firewall rule is broader than a private endpoint model.
- Container image deployment is performed with Azure CLI updates in the release pipeline, so image tags are not the only source of truth in Terraform state.
- The application currently uses the PostgreSQL administrator account by default unless an app-specific database role is created as a post-provision step.
- Custom domain and certificate automation are documented but not fully automated in this MVP baseline.

## Deferred Controls

- Private networking for PostgreSQL and Container Apps
- Dedicated least-privilege application database role fully automated through controlled database provisioning
- WAF or Front Door integration
- SIEM forwarding and alert rules beyond the logging baseline
- Secret rotation automation rather than operator-initiated rotation
- Vulnerability scanning/signing enforcement in the image promotion path

## Operational Recommendations

- Protect Terraform state with RBAC, logging, soft delete, and storage account firewall controls.
- Enable GitHub environment approvals and branch protection on `main`.
- Add ACR image retention and vulnerability scanning policies if your subscription tier supports them.
- Create a dedicated application database user before production go-live.
- Review runtime logs and failed migration job executions after each release.

## Disclosure Note

This repository is designed to align with a practical ISO/IEC 27001-style cloud deployment baseline for an MVP. It does not imply formal certification or a completed ISMS program.
